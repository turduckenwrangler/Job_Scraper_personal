Surface Salary in the Indeed Digest

Capture the pay data python-jobspy already returns for Indeed roles and render it in the Indeed digest — without touching the LinkedIn or biotech pipelines.

2026-05-30 Implementation Plan Target: scrape_jobs.py

Overview

The three watchers each emit a .json / .md / .html digest through one shared path, save_jobs_output(). The Indeed watcher pulls results via python-jobspy, whose result rows include compensation columns — but scrape_indeed_recent() drops them: the job dict it builds (scrape_jobs.py:770–777) keeps only company, title, location, url, date_posted, and ats.

This plan adds an optional salary field, populated only on the Indeed path, and renders it in that digest's Markdown and HTML. Because every renderer reads the field defensively with .get("salary"), the LinkedIn, biotech, and legacy outputs need no change — they simply never set it.

Goal in one line
Stop throwing away salary data we already fetch; show it on Indeed cards. Minimal blast radius, no schema change for sources that lack pay data.

Data flow

Salary is captured once at the Indeed row, normalized to a display string, stored on the job dict, then read by the two Indeed renderers. The JSON file gains one key as a side effect of storing it on the dict.

2. The format_salary() helper

Pure string formatting — no jobspy import, easy to unit-test in isolation:

scrape_jobs.py · new, near _job_identity
def format_salary(min_amount, max_amount, interval) -> str:
    """'$150k–$190k/yr' from jobspy pay fields; '' when no data."""
    def k(v):
        try:
            return f"${int(float(v)) // 1000}k"
        except (TypeError, ValueError):
            return ""
    lo, hi = k(min_amount), k(max_amount)
    if not lo and not hi:
        return ""
    span = f"{lo}–{hi}" if lo and hi and lo != hi else (lo or hi)
    unit = {"yearly": "/yr", "hourly": "/hr"}.get(str(interval).lower(), "")
    return f"{span}{unit}"

Gotchas & open questions

Unverified — jobspy column names
The fields min_amount / max_amount / interval are assumed. requirements.txt pins no version (python-jobspy only), and the schema has shifted across releases. Verify against the installed version before trusting these keys — a wrong name silently yields "" (no salary ever shows, no error).
Risk — triage.html JSON coupling
Adding salary to each job in indeed_jobs.json changes the shape that triage.html fetches and merges at view time. Confirm the dashboard ignores unknown keys (it should — JS reads named fields), and decide whether the cockpit should display salary too, or just the email digest.
Completeness — tests
The repo has no test suite. At minimum add a focused unit test for format_salary() covering: both bounds, one bound, neither, equal bounds, and hourly vs yearly intervals.
Scope — keep it small
In scope: USD, yearly/hourly. Out of scope: currency conversion, locale formatting, per-source salary on LinkedIn (the guest endpoint doesn't expose it reliably). Resist gold-plating.
